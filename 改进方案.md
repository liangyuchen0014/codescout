## 1. 怎样重新设计 CodeScout 的 reward

你的判断基本正确，但更准确地说，问题不在于 reward “不像语言模型 loss 那样可微或平滑”——强化学习的环境奖励本来就不需要可微；真正的问题是：

1. **同一问题的多个 rollout 之间，reward 分辨率较低**；
2. **file、module、entity 三层 reward 高度相关，存在重复计分**；
3. **大量 rollout 可能同时得 0 分或满分，使组内 advantage 接近 0**；
4. **最终结果奖励很难告诉模型：中间哪一次搜索、阅读或跳转真正有用。**

CodeScout 当前使用：

[
R_{\text{loc}}
==============

F1_{\text{file}}
+
F1_{\text{module}}
+
F1_{\text{function}}
]

然后用组内平均 reward 构造 GSPO advantage。论文没有对组内 reward 做标准差归一化，也没有加入 KL 项。

### 一个容易忽略的实现问题：module 和 entity 可能重复计分

从官方实现看，预测位置会被转换成集合后分别计算 F1。对于独立函数：

* module 标识可能是 `file:function`
* entity 标识同样也是 `file:function`

因此，一个独立函数命中时，module 和 entity 两项实际上可能重复奖励同一件事；而类方法则会分别形成 `file:Class` 和 `file:Class.method`。这会造成不同类型实体的 reward 结构不一致，也会放大你所说的“跳变”。([GitHub][1])

## 2. 我建议采用“结果奖励 + 可验证过程奖励 + 轻量效率约束”

总体形式可以设计成：

[
R
=

\alpha R_{\text{outcome}}
+
\beta R_{\text{process}}
+
\gamma R_{\text{ground}}
------------------------

\eta C_{\text{efficiency}}
]

第一版可以从下面的权重开始做消融：

[
R
=

0.70R_{\text{outcome}}
+
0.20R_{\text{process}}
+
0.10R_{\text{ground}}
---------------------

0.03C_{\text{efficiency}}
]

这些数值不是最终最优值，而是较安全的实验起点。核心原则是：**最终定位正确性始终占主要权重，过程奖励只负责改善 credit assignment，不能凌驾于最终结果之上。**

### 2.1 先修正最终结果奖励

建议先把预测位置统一成原子化结构：

[
l=(\text{file},\text{class or None},\text{function or None})
]

然后分别计算：

* 文件定位 F1；
* 类定位 F1，只统计真正的类；
* 函数/方法定位 F1；
* 可选的完整位置 tuple F1。

例如：

[
R_{\text{outcome}}
==================

0.20F1_{\text{file}}
+
0.15F1_{\text{class}}
+
0.45F1_{\text{function}}
+
0.20F1_{\text{tuple}}
]

其中：

[
F1_{\text{tuple}}
=================

F1\big(
(\text{file},\text{class},\text{function})
\big)
]

这样有几个好处：

* 消除独立函数在 module/entity 层的重复计分；
* 文件正确但函数错误时仍有部分奖励；
* 函数完整定位的奖励最高；
* tuple F1 保证模型不能通过分别猜对若干文件名和函数名来“拼凑”高分。

也可以采用 curriculum：

* 训练初期提高 file 权重，帮助模型学会进入正确搜索空间；
* 中后期逐渐提高 function 和 tuple 权重。

例如：

[
\begin{aligned}
\text{early}:&\quad 0.4F_f+0.25F_c+0.25F_u+0.1F_t\
\text{late}:&\quad 0.15F_f+0.15F_c+0.45F_u+0.25F_t
\end{aligned}
]

但不要只对原始 reward 做 sigmoid、开方等变换。它们可以压缩数值，却不会增加 rollout 之间的信息量，也无法解决全 0、全满分的问题。

## 3. “读到正确文件就给奖励”是可行的，但要避免奖励投机

你的 PRM 思路很适合代码定位任务，因为中间过程本身具有大量**可程序化验证**的状态，不一定需要先训练一个神经网络 PRM。

定义在第 (t) 步之前已经真正查看过的 gold 信息覆盖率：

[
C_f(t)
======

\frac{\left|\text{已打开的 gold files}\right|}
{\left|\text{gold files}\right|}
]

类似地定义：

[
C_c(t),\quad C_u(t)
]

分别表示 gold class 和 gold function 的覆盖率。再定义一个状态势函数：

[
\Phi(s_t)
=========

0.2C_f(t)
+
0.3C_c(t)
+
0.5C_u(t)
]

每一步的过程奖励是：

[
r^{\text{process}}_t
====================

\Phi(s_{t+1})-\Phi(s_t)
]

它有三个重要性质：

* 第一次打开正确文件时获得正奖励；
* 重复打开同一文件不会反复刷分；
* 从文件进一步定位到类和函数时，会继续得到增量奖励。

### 什么叫“真正阅读到了正确文件”

不要仅凭工具输出中出现了 gold 文件名就奖励，否则模型可能通过：

```bash
find . -type f
grep -R "keyword" .
```

一次扫出大量路径来刷奖励。

建议只有满足下面条件时才算有效阅读：

* 模型显式调用了 `cat`、`sed`、文件读取工具或 IDE 打开操作；
* 返回内容确实覆盖了目标代码区域；
* 工具输出没有因为长度截断而根本没展示目标位置；
* 搜索命中路径只能获得很小奖励，真正读取文件获得更大程度的奖励。

可以区分：

[
\begin{array}{ll}
\text{搜索结果出现正确文件} & +0.02\
\text{第一次打开正确文件} & +0.06\
\text{阅读到正确类定义区域} & +0.08\
\text{阅读到正确函数代码区域} & +0.14\
\text{最终答案引用了已阅读证据} & +0.05
\end{array}
]

这里最好采用“首次命中”机制，而不是每次访问都加分。

### 不要惩罚所有非 gold 文件阅读

CodeScout 的 gold location 是根据补丁修改区域构造的。论文也承认：这些位置可能漏掉理解问题或完成修复所必需的其他代码。因此，模型阅读一个非 gold 文件，不代表它走错了。([arXiv][2])

所以过程 reward 应当是：

* 对读取 gold 位置给予明确正奖励；
* 对非 gold 阅读通常不给分，而不是直接扣分；
* 只对明显无效行为扣分，如重复执行相同命令、反复读取同一区域、输出极大量无关文本。

## 4. 仅把过程奖励加到最终 scalar 中，还不是真正的“及时 credit assignment”

这是非常关键的一点。

CodeScout 当前采用 sequence-level GSPO。假如最后把所有步骤奖励相加：

[
R_{\text{total}}
================

R_{\text{final}}
+
\sum_t r_t
]

然后仍然给整个序列一个统一 advantage，那么虽然 reward 数值更丰富了，但序列里：

* 正确打开文件的那一步；
* 无意义搜索的那一步；
* 最终输出答案的那一步；

仍然共享同一个 advantage。

更彻底的做法是使用**按 assistant turn 或工具动作分配的 advantage**：

[
A_{i,t}
=======

\alpha A^{\text{terminal}}*i
+
\beta
\left(
\Delta\Phi*{i,t}
----------------

\operatorname{mean}*{j\in G}\Delta\Phi*{j,t}
\right)
-------

\eta C_{i,t}
]

其中：

* 最终定位 reward 可以作用于整条轨迹；
* (\Delta\Phi_{i,t}) 只作用于产生该工具动作的 assistant turn；
* 重复查询和超长输出的 cost 只作用于对应步骤。

训练时，只把这个 turn-level advantage 施加到该轮 assistant token 上。这样模型才能真正学到：

> 这个搜索模式更容易找到正确文件，而不是整段轨迹碰巧得了高分。

这可以成为一个比较明确的论文创新点：

**Turn-level verifiable process reward for repository-level code localization。**

## 5. 是否需要训练一个真正的神经网络 PRM

我建议分两阶段。

### 第一阶段：规则型、可验证的过程奖励

代码定位有天然监督信号：

* 文件路径是否正确；
* 是否看到目标 AST 节点；
* 阅读范围是否覆盖目标函数；
* 最终位置是否来自之前观察到的证据。

这些信号比通用推理 PRM 更可靠，也更不容易出现 reward hacking。

### 第二阶段：训练 trajectory-prefix value model

对于轨迹前缀 (h_t)，从该前缀继续采样 (K) 个后续轨迹，用最终 localization reward 估计：

[
V(h_t)
======

\frac{1}{K}
\sum_{k=1}^{K}
R_{\text{outcome}}^{(k)}
]

然后训练一个模型预测：

> 当前已经搜索到这些文件、读过这些代码以后，最终成功定位的概率或期望 F1 是多少。

使用：

[
r^{\text{PRM}}_t
================

V(h_{t+1})-V(h_t)
]

作为额外奖励。

不过它的权重应该较小，例如占总 reward 的 5%～10%，并且不能替代可验证的最终 reward。否则模型可能学会利用 PRM 的偏差，而不是真正提高定位能力。

另外需要纠正一个概念：DeepSeek-R1 确实使用了多组件 reward，例如正确性和格式奖励，但官方论文明确表示其大规模 RL 没有使用神经网络 outcome reward model 或 process reward model；论文还将 PRM 列为尝试过但存在步骤标注困难和 reward hacking 的方法。([arXiv][3])

所以你可以借鉴的是“多目标奖励”，但不宜写成“DeepSeek-R1 使用 PRM”。

## 6. 高上限效应还要从采样和 advantage 上解决

过程 reward 能缓解全 0 问题，但对于已经完全学会的简单样本，所有 rollout 都满分，组内 advantage 仍然接近 0。

建议记录每个 prompt 最近若干次 rollout 的成功率 (p_q)，采样权重设为：

[
w(q)
\propto
4p_q(1-p_q)+\epsilon
]

这样：

* (p\approx 0.5) 的样本采样概率最高；
* 所有 rollout 都成功的简单样本逐渐降权；
* 所有 rollout 都失败的超难样本也适当降权；
* 仍保留一个 (\epsilon)，防止它们彻底消失。

也可以按组内 reward 方差采样：

[
w(q)\propto \operatorname{Var}(R_q)+\epsilon
]

此外，CodeScout 附录中的实验显示：

* GSPO 原始设置的 file F1 较高；
* 标准差归一化和 Dr.GRPO 在 module/function F1 上反而更高；
* 不同 advantage 归一化方式对不同层级指标有明显影响。([arXiv][2])

因此，除了 reward 本身，建议同时消融：

* mean-only advantage；
* mean/std advantage；
* rank-based group advantage；
* reward component 分别归一化后再求和。

其中 rank-based advantage 对离群值和不同 reward 尺度通常更稳定。例如组内第 (k) 名可映射到：

[
A_k
===

2\frac{\operatorname{rank}(R_k)-1}{G-1}-1
]

不过并列 reward 很多时，rank-based 方法仍然无能为力，因此必须和过程 reward 配合。

# 7. 基于 Qwen3 训练还是基于 CodeScout-4B 继续训练

你的理解有一半正确：

* **从 CodeScout-4B 开始，很可能更容易获得较高的绝对指标；**
* **但使用 CodeScout-4B 本身并不构成学术不端。**

CodeScout-4B 官方模型卡标明它直接从 `Qwen3-4B-Instruct-2507` 训练而来，采用 Apache-2.0 许可证；Qwen3 对应模型同样使用 Apache-2.0，CodeScout 代码仓库则使用 MIT 许可证。只要遵循许可证、正确引用并清楚披露初始化模型，继续训练 CodeScout-4B 在学术上完全可以接受。([Hugging Face][4])

真正的问题不是学术不端，而是**实验结论能否被清楚归因**。

## 从 CodeScout-4B 开始的局限

CodeScout-4B 已经使用原始 localization reward，在约 1.6K 个训练实例上进行了 200 步 GSPO 训练。([arXiv][2])

如果你直接从它继续训练，最后指标提升，审稿人可能提出：

* 提升是新 reward 导致的，还是单纯多训练了若干步？
* 是新训练数据带来的，还是 reward 带来的？
* 原始模型继续用旧 reward 训练同样步数，是否也会提高？
* 新 reward 是否只有在 CodeScout 已经掌握代码定位能力后才有效？

因此必须加入：

[
\text{CodeScout-4B}
+
\text{原 reward 继续训练}
]

作为严格对照，而不能只比较：

[
\text{原始 CodeScout-4B}
\quad\text{vs.}\quad
\text{CodeScout-4B + 新 reward}
]

## 最适合发论文的实验设计

建议采用双轨实验。

### 主实验：从相同通用模型出发，验证 reward 的因果效果

| 初始化                    | 训练方法                         |
| ---------------------- | ---------------------------- |
| Qwen3-4B-Instruct-2507 | 原始 CodeScout reward          |
| Qwen3-4B-Instruct-2507 | 你的新 outcome reward           |
| Qwen3-4B-Instruct-2507 | 新 outcome + process reward   |
| Qwen3-4B-Instruct-2507 | 新 reward + turn-level credit |

所有实验保持：

* 相同训练数据；
* 相同训练步数；
* 相同 rollout 数量；
* 相同最大 turn 数；
* 相同推理预算；
* 至少 3 个随机种子。

这组实验回答的是：

> 在完全相同的基础模型和计算预算下，新 reward 是否比 CodeScout 原 reward 更有效？

这是论文中最干净、最有说服力的结论。

### 扩展实验：从 CodeScout-4B 继续训练，追求最佳指标

| 初始化          | 继续训练方式                    |
| ------------ | ------------------------- |
| CodeScout-4B | 原始 reward                 |
| CodeScout-4B | 新 reward                  |
| CodeScout-4B | 新 reward + process credit |

这组实验回答的是：

> 新 reward 能否进一步提升已经特化的代码定位模型，并获得更高绝对指标？

如果最终最高分来自这一组，完全可以作为论文的 best model，只需明确将其描述为：

> continued reinforcement learning from CodeScout-4B

而不是声称整个模型完全由你从 Qwen3 独立训练得到。

## 8. 我的具体建议

在只能选择一条路线时：

* **目标是验证方法、形成最干净的论文贡献：选择 Qwen3-4B；**
* **目标是尽量冲高 benchmark 指标：选择 CodeScout-4B；**
* **比较理想的论文方案：Qwen3 做主实验，CodeScout-4B 做 continued-RL 扩展实验。**

训练数据还应保持 repository-level 隔离，不能让同一仓库分别出现在训练集和评测集中。CodeScout 原论文也明确进行了训练仓库与评测仓库的去重；你的新增训练数据和 process-reward 轨迹也应遵循同样标准。([arXiv][2])

最终可以把论文贡献凝练成三点：

1. **重新定义无重复计分的层级代码定位 outcome reward；**
2. **提出基于文件、类、函数实际阅读覆盖率的可验证过程奖励；**
3. **提出 turn-level GSPO credit assignment 和难度自适应采样，缓解全 0、并列和上限样本问题。**

这比单纯修改 file/module/function 三项权重，更容易形成完整且可论证的方法创新。

## 1. 论文中的 100、200、300 个 training step 是怎么决定的？

### 先看论文给出的实际训练配置

| 模型             | RL 起点                  | 训练实例数 | 每步实例数 | 每实例 rollout 数 | Training steps |
| -------------- | ---------------------- | ----: | ----: | ------------: | -------------: |
| CodeScout-1.7B | CodeScout-1.7B-RFT     |   800 |     8 |             8 |            100 |
| CodeScout-4B   | Qwen3-4B-Instruct-2507 | 1,600 |     8 |             8 |            200 |
| CodeScout-14B  | Qwen3-14B              | 9,600 |    32 |             4 |            300 |

这些数字满足：

[
\text{training steps}
=====================

\frac{\text{训练实例数}}{\text{train batch size}}
]

具体为：

[
800/8=100
]

[
1600/8=200
]

[
9600/32=300
]

因此，**最合理的解释不是作者观察曲线收敛后，在 100/200/300 步早停，而是作者先为不同规模的模型分配了不同大小的训练数据子集，然后让模型基本遍历一遍这些实例**。论文只明确报告了这些数据量和步数，没有说明进行过基于验证集的 early stopping，也没有解释为什么恰好给 4B 1.6K 条、14B 9.6K 条。([arXiv][1])

换句话说，训练步数更像是：

> 数据预算和算力预算确定以后自然得到的结果。

而不是：

> 根据 reward 曲线找到最优步数以后决定的结果。

---

### 一个 training step 实际用了多少 rollout？

在 CodeScout 中，一个训练实例就是一个 issue。对同一个 issue 采样多个 rollout，构成 GSPO 的 group。

所以每一步名义上的轨迹数是：

#### CodeScout-1.7B

[
8\text{ issues}\times 8\text{ rollouts}=64\text{ trajectories}
]

总 rollout 数约为：

[
100\times64=6400
]

#### CodeScout-4B

[
8\times8=64
]

总 rollout 数约为：

[
200\times64=12800
]

#### CodeScout-14B

[
32\times4=128
]

总 rollout 数约为：

[
300\times128=38400
]

这里说“约”是因为它使用异步训练：trajectory 最多可以来自落后 4 个 step 的 checkpoint，而且参数更新后会终止尚未完成的 inference request。因此实际成功记录的 rollout 数可能和理论乘积稍有不同。([arXiv][1])

这也说明，**不同模型之间不能简单比较 training step 数**：

* 4B 每步约 64 条轨迹；
* 14B 每步约 128 条轨迹；
* 14B 不仅步数更多，每步使用的轨迹也更多。

从名义 rollout 预算看，14B 大约是 4B 的三倍：

[
38400/12800=3
]

---

## 2. 你的曲线观察是正确的，但还要注意一个问题

附录 B 中，作者明确给出了这样的结论：

* CodeScout-1.7B 在前约 20 步快速提升，此后基本饱和；
* CodeScout-4B 和 14B 到训练结束时仍没有明显饱和；
* 作者认为继续使用更多 GitHub issue 训练 4B 和 14B，可能继续提高定位性能。([arXiv][1])

从图上看：

* 1.7B 大约在 20～30 步达到较高点，随后主要在波动；
* 4B 的 function-level F1 到第 200 步仍总体上升；
* 14B 的 function-level F1 到第 300 步也仍有明显上升趋势。

所以可以得出一个比较确定的判断：

> **200 步和 300 步不是 4B、14B 的收敛步数，而只是此次实验的训练预算终点。**

作者自己也承认这两个模型“no visible signs of saturation”。

---

### 但不能直接说“1.7B 最佳 checkpoint 是第 30 步”

附录里的曲线是**训练 batch 上的在线 reward**，并且使用窗口为 16 个 training step 的滑动平均。([arXiv][1])

每个 step 使用的是不同 issue，因此曲线变化同时受到两个因素影响：

[
\text{观测 reward 变化}
===================

\text{模型能力变化}
+
\text{当前 batch 难度变化}
]

例如：

* 第 30 步可能碰巧包含较简单的 issue；
* 第 60 步可能包含较困难的 issue；
* 曲线下降不一定意味着模型真实性能下降。

因此，图 9 能说明的是：

> 1.7B 后续训练的边际收益很小。

但它不能严格证明：

> 第 25 或第 30 步一定是测试集性能最高的 checkpoint。

要决定最优训练轮数，应当定期在**固定验证集**上评测，而不是只看在线训练 reward。

## 4. 你重新设计 reward 后，训练多久比较合理？

不要直接规定“也训练 200 步”。应该区分两个实验目的。

### 实验 A：公平证明新 reward 更好

原始 reward 和新 reward 保持完全相同的：

* 初始化模型；
* 训练 issue；
* issue 顺序；
* batch size；
* 每个 issue 的 rollout 数；
* 最大 turns；
* 最大 token 数；
* learning rate；
* 总 training steps。

例如都从 Qwen3-4B 开始，在相同的 1.6K issue 上训练 200 步。

这样才能回答：

> 在相同轨迹预算下，新 reward 是否比旧 reward 学得更好？

不能因为新 reward 训练了 400 步，而旧 reward 只训练 200 步，然后把提升归因于 reward。

### 实验 B：追求你的最佳模型

再扩大到更多未见过的训练 issue，例如：

* 3.2K issue，约 400 步；
* 6.4K issue，约 800 步；
* 或者使用更大一部分 39K 训练集。

论文一共过滤出了 39K 个训练实例，但 4B 只用了其中 1.6K，14B 只用了 9.6K。([arXiv][1])

考虑到 4B 曲线尚未饱和，我会更建议：

> 增加不同的训练 issue，而不是在同一批 1.6K issue 上反复训练多轮。

这样更不容易记忆特定 repository 或 issue 模式。

# 5. “新 outcome + process reward”是什么意思？

这里的重点是：**只修改 reward，仍然采用 CodeScout 原来的 sequence-level GSPO。**

## 原始 CodeScout

完成整个 rollout 后，得到一个最终奖励：

[
R_i^{\text{old}}
================

F1_{\text{file}}
+
F1_{\text{module}}
+
F1_{\text{function}}
]

同一个 issue 有 (G) 个 rollout，CodeScout 计算：

[
A_i
===

R_i-\frac{1}{G}\sum_{j=1}^{G}R_j
]

这个 (A_i) 是**整条 trajectory 的 advantage**。论文使用 sequence-level importance ratio，因此整条轨迹中的所有模型生成 token，共享同一个 advantage。([arXiv][1])

---

## 新 outcome reward

这是对最终提交答案的评价，例如：

[
R_i^{\text{out}}
================

0.2F1_{\text{file}}
+
0.15F1_{\text{class}}
+
0.45F1_{\text{function}}
+
0.2F1_{\text{tuple}}
]

它仍然只在 agent 调用 `localization_finish` 后计算。

它回答的是：

> 最终提交的位置对不对？

---

## Process reward

在 rollout 中间，根据模型的搜索和阅读行为产生奖励。例如：

* 搜索结果首次出现正确文件：( +0.02 )
* 首次真正打开正确文件：( +0.05 )
* 查看范围覆盖正确类：( +0.08 )
* 查看范围覆盖正确函数：( +0.12 )
* 重复查看相同内容：( -0.01 )

也可以用增量覆盖率定义：

[
\Phi(s_t)
=========

0.2C_{\text{file}}(t)
+
0.3C_{\text{class}}(t)
+
0.5C_{\text{function}}(t)
]

其中 (C(t)) 表示截至第 (t) 轮已经实际读取的 gold 位置比例。

每一步过程奖励为：

[
r^{\text{proc}}_{i,t}
=====================

\Phi(s_{i,t+1})-\Phi(s_{i,t})
]

因此重复读取已经找到的同一文件不会再次得到奖励。

---

## 最后仍然压成一个 trajectory reward

[
R_i
===

R_i^{\text{out}}
+
\lambda\sum_{t=1}^{T_i}r^{\text{proc}}_{i,t}
]

然后仍按原始 CodeScout 的方法计算：

[
A_i=R_i-\operatorname{mean}(R)
]

这就是我所说的：

> **新 outcome + process reward。**

它的特点是：

* reward 更稠密；
* 找到正确路径但最终输出略错的 rollout，不再只能得到很低奖励；
* 不需要大幅修改 GSPO loss；
* 主要修改 reward evaluator 和 trajectory parser。

但它仍有一个局限：

> 中间所有 turn 最终还是共享同一个 trajectory-level advantage。

---

## 举一个例子

一个 rollout 有三轮：

### Turn 1

```bash
rg "timezone" django/db
```

发现了正确文件：

[
r_1^{\text{proc}}=0.05
]

### Turn 2

打开了一个无关文件：

[
r_2^{\text{proc}}=0
]

### Turn 3

读取到了正确函数：

[
r_3^{\text{proc}}=0.15
]

最终答案得分：

[
R^{\text{out}}=0.8
]

若 (\lambda=0.5)，则：

[
R=0.8+0.5(0.05+0+0.15)=0.9
]

在普通 sequence-level GSPO 中，Turn 1、Turn 2、Turn 3 的所有 token 都共享基于 (0.9) 得到的同一个 advantage。

所以模型只能学到：

> 这整条轨迹总体不错。

但不能十分精确地学到：

> Turn 1 和 Turn 3 是有效行为，Turn 2 没什么价值。

---

# 6. “新 reward + turn-level credit”是什么意思？

它不仅修改 reward，还修改**reward 被分配给哪些 token**。

原始 CodeScout 把多轮 trajectory 合并成一条训练序列，并使用一个 sequence-level advantage；系统消息、用户消息和工具返回内容会被 mask，只训练模型自己生成的内容。([arXiv][1])

turn-level credit 则为每一个 assistant turn 构造不同的 advantage：

[
A_{i,1},A_{i,2},\ldots,A_{i,T}
]

并且：

* Turn 1 的 token 只使用 (A_{i,1})；
* Turn 2 的 token 只使用 (A_{i,2})；
* Turn 3 的 token 只使用 (A_{i,3})。

---

## 一种简单稳定的构造方法

先保留最终 outcome 的 trajectory-level advantage：

[
A_i^{\text{out}}
================

## R_i^{\text{out}}

\operatorname{mean}_{j\in G}R_j^{\text{out}}
]

再为每一轮计算局部 process advantage：

[
A_{i,t}^{\text{proc}}
=====================

## r_{i,t}^{\text{proc}}

b_t
]

其中 (b_t) 可以是同一 issue 下各 rollout 在第 (t) 轮过程奖励的平均值。

最终：

[
A_{i,t}
=======

\alpha A_i^{\text{out}}
+
\beta A_{i,t}^{\text{proc}}
]

这样：

* 最终定位正确，整条轨迹都会获得 outcome 信号；
* 找到正确文件的具体 turn 会额外得到 process 正信号；
* 无效或重复搜索的 turn 不会因为其他轮做得好而获得同样高的奖励。

---

## 更完整的 return-to-go 方法

为了让前面的搜索动作也能获得“它帮助了后续成功”的信用，可以定义：

[
G_{i,t}
=======

\gamma^{T-t}R_i^{\text{out}}
+
\sum_{k=t}^{T}
\gamma^{k-t}r_{i,k}^{\text{proc}}
]

再计算：

[
A_{i,t}
=======

G_{i,t}-b(q,t)
]

这里 (G_{i,t}) 是从第 (t) 轮开始的 return-to-go。

以刚才的三轮轨迹为例，假设暂时取 (\gamma=1)：

[
G_1=0.05+0+0.15+0.8=1.0
]

[
G_2=0+0.15+0.8=0.95
]

[
G_3=0.15+0.8=0.95
]

如果 Turn 2 是明显的重复无效行为，还可以给予负成本，那么其 advantage 会进一步降低。

---

## 两种方法的核心区别

| 方法                           | 是否增加过程奖励 | advantage 粒度      | 需要修改什么                                 |
| ---------------------------- | -------: | ----------------- | -------------------------------------- |
| 原始 CodeScout                 |        否 | 整条 sequence       | 无                                      |
| 新 outcome + process reward   |        是 | 整条 sequence       | reward evaluator                       |
| 新 reward + turn-level credit |        是 | 每个 assistant turn | reward evaluator + loss/advantage/mask |

可以把第一种理解为：

> **哪条完整轨迹更好？**

把第二种理解为：

> **这条轨迹中的哪一步做得好？**

---

## 7. 这两种方法之间的推荐递进关系

不建议一开始同时大改 reward 和 GSPO loss，因为一旦结果异常，很难判断问题来自哪里。

建议按以下顺序做消融：

### Baseline

[
F1_{\text{file}}+F1_{\text{module}}+F1_{\text{function}}
]

使用原始 sequence-level GSPO。

### 实验 1：新 outcome

只修正最终层级 reward，不加入过程奖励。

### 实验 2：新 outcome + process reward

将过程奖励累加到最终 scalar 中，但保持原始 sequence-level GSPO。

这能验证：

> 稠密的过程监督本身是否有效。

### 实验 3：新 reward + turn-level credit

相同 reward，不再将它全部压成一个 sequence scalar，而是对每个 turn 分配 advantage。

这能验证：

> 精确的 temporal credit assignment 是否能在同样 reward 下进一步提高性能。

因此，“实验 2”和“实验 3”不是两套完全不同的 reward。更准确地说：

* 实验 2主要是**奖励设计创新**；
* 实验 3是在实验 2基础上，再加入**credit assignment 创新**。

对你当前的复现阶段，我建议先实现“新 outcome + process reward”。它和现有 CodeScout/SkyRL 训练框架的兼容性最高；确认 reward 有效后，再修改为 turn-level advantage。
