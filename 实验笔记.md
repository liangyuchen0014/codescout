
### 1. `-w step-wise` 开关是什么意思？

在scripts\run_async_training_4b.sh 中，-w表示 step-wise 开关。

在构建智能体（Agent）的强化学习中，通常要让模型在环境中生成轨迹（比如代码搜索：思考 -> 查目录 -> 读代码 -> 输出结果）。

* **关闭时（当前默认 False - 轨迹级 Trajectory-level）**：模型会连贯地与环境交互，直到整个任务完成或达到最大轮数（`max_turns`）。最终拿整条序列（所有交互内容）及其最终的总体奖励（Outcome Reward）去更新模型。
* **开启时（Step-wise True - 逐块/过程级 Process-level）**：它会变为步进级别的生成和训练。也就是在智能体的**每一个动作回合（Turn / Step）**都可以进行干预或奖励评估（Process Reward Model, PRM）。这种模式允许在每一步执行后，对该步的工具调用对错直接给出细粒度反馈并用于训练，对于长链条推理或多步 Agent 任务（如代码定位）效果通常更好，可以防止“中间走错但最终误打误撞猜对”引发的奖励分配虚高问题。

>  **感觉可以开启step-wise，然后如果Agent调用read_file更贴近目标答案所在函数的调用链的话，就给这步工具调用打更高的分数。**

### 2. 为什么训练时需要同时分配“推理引擎”和“训练引擎”？

当代高效的大模型强化学习框架（如其底层依赖的 Ray / vLLM + FSDP 架构）使用了**计算分离异步架构（Decoupled & Async Architecture）**。因为 LLM 的“生成”与“训练”在底层的计算和显存特性上完全不同：

* **推理引擎（Inference Engine / Actor）**：
  * **职责**：与环境交互、采样 Rollout，生成海量的动作轨迹，也就是产生经验数据。
  * **难点与优化**：自回归生成非常慢。为了加速，它使用了高度优化的引擎（如脚本中的 `generator.backend=vllm`），专门为了管理 KV Cache（PagedAttention）进行极致的吞吐加速，但不保存梯度计算图。
* **训练引擎（Training Engine / Learner）**：
  * **职责**：接收推理引擎产生的数据，计算 Loss（比如 GRPO 损失、KL 散度损失等），反向传播计算梯度并更新模型权重。
  * **难点与优化**：需要保存大量的中间激活状态和优化器（Optimizer）状态。所以它使用传统的训练流框架（如脚本中的 `trainer.strategy=fsdp2`），用于多显卡分布式梯度同步。

**同步协作机制**：
这两套引擎常驻在显卡内存中并发运行。**推理引擎**疯狂地生产数据（产生轨迹），丢到共享池里；**训练引擎**不断从池里拿数据做反向传播。当训练引擎更新完策略权重后，使用极高带宽的网络或显存拷贝（如脚本里的 `generator.weight_sync_backend=nccl`）把最新权重广播给前方的推理引擎。
如果不这样分离，而是统使用 PyTorch 一步步生成再训练，生成过程没有 vLLM 这种吞吐加速，整个强化学习循环会慢几个数量级。分库虽然吃显存，但保证了整体的执行训练的最大算力和显卡利用率拉满。
